WHAT YOU ARE SETTING UP
------------------------------------------------------------------------------

You will use Google Gemini CLI -- a terminal-based AI coding agent that
works natively on Windows. No proxies, no wrappers, no translation layers.

  Mechanical tasks  -->  Gemini 2.0 Flash   (fast, cheap)
  Real coding       -->  Gemini 2.5 Pro     (reasoning, architecture)

The session launcher scripts select the model automatically.
You never type model names by hand.

The workflow mirrors Claude Code exactly:
  GEMINI.md   = CLAUDE.md   (project context and rules)
  superpowers = superpowers  (same extension, works on both)
  MCP servers = MCP servers  (filesystem / memory / sequential-thinking)
  sessions    = sessions     (same launcher pattern)

NEW IN v3 -- what the scripts now do automatically:
  - The setup script GENERATES <project>-sessions.ps1 with all modules
    pre-filled. You do NOT copy or edit g_sessions_template.ps1 manually.
  - Every session writes a local checkpoint to .checkpoints\ckNN.md
  - Every new session auto-resumes from the latest checkpoint
  - Every session end offers git commit + push in one keystroke

------------------------------------------------------------------------------
PART A -- ONE-TIME MACHINE SETUP
Do this once on each new machine.
------------------------------------------------------------------------------

STEP A1 -- Unblock PowerShell scripts (do this first, every time)

  Windows blocks downloaded .ps1 files by default.
  In PowerShell, navigate to the folder where you placed the scripts:

    cd D:\staging\_gemini-setup
    Get-ChildItem -Filter *.ps1 | Unblock-File

  Then allow scripts to run:
    Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

  Type Y and press Enter when prompted.
  You only need to do this ONCE per machine, but always unblock new .ps1 files.


STEP A2 -- Install prerequisites (in this order)

  1. Node.js LTS
     https://nodejs.org
     Download the LTS installer. Run it. Accept all defaults.
     This also installs npm automatically.

     Verify (open a NEW PowerShell after install):
       node --version
       npm --version
     Both must print version numbers before continuing.

  2. Git
     https://git-scm.com
     Download and run installer. Accept all defaults.

     Verify:
       git --version

  3. Python 3.x
     https://python.org
     Download installer. IMPORTANT: check "Add Python to PATH" on first screen.

     Verify:
       python --version


STEP A3 -- Get your Gemini API key (Pro plan)

  1. Go to: https://aistudio.google.com/app/apikey
  2. Sign in with your Google account.
  3. Click "Create API Key". Select your Pro project or create one.
  4. Copy the key -- it starts with: AIzaSy...

  Set it permanently in Windows (run in PowerShell, replace YOUR_KEY_HERE):
    [System.Environment]::SetEnvironmentVariable("GEMINI_API_KEY","YOUR_KEY_HERE","User")

  Close PowerShell completely. Open a new PowerShell window.

  Verify:
    echo $env:GEMINI_API_KEY
  You must see your key printed. If you see nothing, redo this step.


STEP A4 -- Run the generic bootstrap script

  Place your _gemini-setup folder at: D:\staging\_gemini-setup\

  In PowerShell:
    cd D:\staging\_gemini-setup
    .\g_setup_generic.ps1

  This script installs and configures everything globally, once:
    - Checks all prerequisites (stops with clear error if anything is missing)
    - Installs Gemini CLI globally via npm
    - Installs MCP servers: filesystem, memory, sequential-thinking
    - Writes global ~/.gemini/settings.json with MCP and model config
    - Writes global ~/.gemini/GEMINI.md with workflow rules
    - Installs the superpowers extension (brainstorm / plan / execute)

  If any step shows FAIL, fix it before continuing.
  This script is safe to re-run as many times as needed.


STEP A5 -- Verify the machine setup works

  Test 1 -- Gemini CLI responds:
    gemini -p "reply with exactly: gemini ok"
  Expected output: gemini ok

  Test 2 -- Flash model works:
    gemini --model gemini-2.0-flash -p "reply with exactly: flash ok"
  Expected output: flash ok

  Test 3 -- Superpowers extension works:
    gemini
    (inside the session, type): superpowers brainstorm
  Expected: Gemini asks clarifying questions about your project.
  Type /quit to exit.

  Fixes:
    Test 1 fails  --> echo $env:GEMINI_API_KEY must not be empty. Redo A3.
    Test 3 fails  --> inside gemini: /extensions install https://github.com/obra/superpowers

  YOU ARE NOW SET UP. Part A is done once per machine.

------------------------------------------------------------------------------
PART B -- STARTING A NEW PROJECT
Do this once per project. Takes about 15 minutes.
------------------------------------------------------------------------------

STEP B1 -- Create your PRD.md (MANDATORY -- never skip this)

  Every project starts with a Product Requirements Document.
  The masterprompt.txt file makes this easy.

  1. Open masterprompt.txt from D:\staging\_gemini-setup\
  2. Copy its ENTIRE contents (Ctrl+A, then Ctrl+C)
  3. Go to https://gemini.google.com
     Click the model selector and choose Gemini 2.5 Pro
  4. Paste into the chat and press Enter
  5. Gemini will interview you ONE SECTION AT A TIME (7 sections total)
     Answer each section fully before Gemini moves on.
     Take your time. The quality of your PRD determines all code output.
  6. After Section 7, Gemini produces the complete PRD.md content
  7. Copy the entire PRD output (Ctrl+A, Ctrl+C in the chat)
  8. Create folder: D:\staging\YOUR-PROJECT-NAME\
  9. Create file:   D:\staging\YOUR-PROJECT-NAME\PRD.md
  10. Paste and save.


STEP B2 -- Configure the project setup script

  1. Copy g_setup_project_template.ps1 from D:\staging\_gemini-setup\
     Rename the copy to: setup-YOUR-PROJECT.ps1
     You can place it anywhere; keeping it in _gemini-setup is fine.

  2. Open setup-YOUR-PROJECT.ps1 in any text editor.
     Fill in ONLY the 5 variables at the very top of the file:

     $PROJECT_NAME  = "your-project-name"
       Use hyphens not spaces. Examples: "jyotish-ai", "resume-builder", "fip"

     $PROJECT_ROOT  = "D:\staging\your-project"
       Full path where your code will live.
       The folder will be created automatically if it does not exist.

     $STACK         = "Python 3.11, FastAPI, PostgreSQL, Docker"
       Comma-separated list of your main technologies.

     $MODULES       = @("auth", "api", "db")
       One entry per module from your PRD phases.
       These become your session names AND your app\ subfolder names.
       Use lowercase, no spaces, no hyphens.
       Example for a 4-module project: @("ingest", "parser", "api", "dashboard")

     $PHASE_CURRENT = 0
       Leave as 0 for any new project.

  3. Save the file.

  4. Make sure PRD.md is already in $PROJECT_ROOT (from Step B1).
     The script hard-stops with fix instructions if PRD.md is missing.


STEP B3 -- Run the project setup script

  In PowerShell:
    cd D:\staging\_gemini-setup      (or wherever you saved the file)
    .\setup-YOUR-PROJECT.ps1

  The script runs 8 steps and shows OK or FAIL for each:

    [0/8] Checks PRD.md exists -- hard stops with fix instructions if missing
    [1/8] Creates all folders:
            tasks\  docs\  tests\  .checkpoints\
            app\auth\  app\api\  app\db\  (one per module you listed)
    [2/8] Writes GEMINI.md  (project context, stack, module scope rules)
    [3/8] Writes TOOLS.md   (model reference, session commands)
    [4/8] Writes SKILLS.md + .gemini\skills\ files (prd-parse, task-create)
    [5/8] Creates tasks\TASK-000-repo-init.md
    [6/8] Writes .gemini\settings.json (MCP servers scoped to this project)
    [7/8] GENERATES D:\staging\your-project\YOUR-PROJECT-sessions.ps1
            Module sessions are pre-filled from your $MODULES list.
            prd-read and debug sessions are included automatically.
            Checkpoint write, resume, and git push are built in.
            You do NOT need to copy or edit g_sessions_template.ps1.
    [8/8] Copies PRD-read prompt to clipboard and launches gemini --model Pro

  After the script completes:
    Open YOUR-PROJECT-sessions.ps1 in $PROJECT_ROOT.
    Find each module session block. Fill in the "Key facts" bullet points.
    These are things only you know: env variable names, API endpoints,
    third-party services, known gotchas for that module.
    Everything else (model, task ID, PDCA rule, SESSION-SUMMARY rule) is
    already filled in.


STEP B4 -- Complete the PRD read session (MANDATORY -- always first)

  The setup script launches gemini automatically with the PRD-read prompt
  already on your clipboard.

  When gemini opens:
    1. Paste (Ctrl+V)
    2. Type: superpowers brainstorm
    3. Gemini reads your PRD.md and proposes all TASK-XXX stubs
    4. Review the list. Tell Gemini to proceed when happy.
    5. Gemini creates all tasks\TASK-XXX-slug.md files
    6. At the end, Gemini types:
         SESSION-SUMMARY: <paragraph of what was done, decisions, next task>
       Copy that paragraph.
    7. Type /quit

  After /quit, the launcher automatically:
    - Writes .checkpoints\ck01.md
    - Asks: "Commit and push to GitHub? [Y/n]"
    - Type Y to commit everything and push.

  Open .checkpoints\ck01.md and paste the SESSION-SUMMARY text
  into the "What was completed" section.

  DO NOT write any code in this session. PRD read is planning only.
  DO NOT skip this session. All subsequent sessions depend on the task files.

------------------------------------------------------------------------------
PART C -- DAILY WORKFLOW
Run every session from here onwards.
------------------------------------------------------------------------------

STEP C1 -- Always use the session launcher

  Never run bare gemini from your project folder.
  Always use the launcher:

    cd D:\staging\your-project
    .\your-project-sessions.ps1 -Session list
    .\your-project-sessions.ps1 -Session auth
    .\your-project-sessions.ps1 -Session debug


STEP C2 -- What happens automatically on every session

  ON LAUNCH:
    1. Launcher checks .checkpoints\ for the latest ckNN.md
    2. If a checkpoint exists, it is prepended to the session prompt
       The banner shows: RESUME: checkpoint injected into prompt
    3. Full prompt is copied to clipboard
    4. gemini launches with the correct model for that session

  AFTER /quit:
    5. Launcher writes the next checkpoint (.checkpoints\ck02.md, ck03.md...)
    6. Launcher asks: "Commit and push to GitHub? [Y/n]"
    7. Type Y -- it runs git add -A, git commit, git push origin <branch>

  You paste at the start and press Y at the end. Everything else is handled.


STEP C3 -- Inside the session

  When gemini opens:
    1. Paste (Ctrl+V) -- includes checkpoint resume block if any
    2. Type: superpowers brainstorm
    3. Follow PDCA: Plan --> show me --> approval --> Do --> Check --> done

  Before /quit, ask Gemini:
    "Please give me a SESSION-SUMMARY paragraph."
  Copy the paragraph. After /quit, paste it into the new checkpoint file
  where it says: [Paste SESSION-SUMMARY from Gemini here]


STEP C4 -- Context discipline (CRITICAL)

  One session = one module = one file scope. No exceptions.
  Do not ask Gemini to touch files outside the declared module.

  If session gets confused or is running very long:
    /chat save mysession
    /quit
    Relaunch with launcher -- checkpoint will auto-resume cleanly.

  Use filesystem MCP to read files inside the session.
  Never paste entire file contents into the chat.


STEP C5 -- Model switching inside a session

  The launcher assigns the right model automatically.
  To override mid-session:
    /model gemini-2.5-pro
    /model gemini-2.0-flash

  Flash for mechanical. Pro for reasoning. When in doubt, use Pro.


STEP C6 -- Git discipline

  Never commit directly to main.
  Always work on feature branches: feature/TASK-XXX
  Commit message format: [TASK-XXX] verb: what changed
  The launcher generates this message automatically at session end.
  Create your branch before starting a module session:
    git checkout -b feature/TASK-001

PART D -- MODEL SELECTION REFERENCE
------------------------------------------------------------------------------

  Use FLASH (gemini-2.0-flash) for:
    Config files, .env, Dockerfiles, docker-compose
    Schema definitions (SQL, Pydantic models, JSON schemas)
    CRUD endpoint boilerplate
    Renaming, moving, restructuring files
    Tests for already-working code
    Any purely mechanical transformation

  Use PRO (gemini-2.5-pro) for:
    Business logic and complex algorithms
    Third-party API integrations
    Debugging tricky errors
    Architecture decisions
    LLM calls, vector search, embeddings
    Docker wiring across multiple services
    Any task where Flash failed or got confused

  The sessions file assigns the correct model per module automatically.
  Odd-numbered module sessions use Flash; even-numbered use Pro.
  Override any session by changing the "model" line in the sessions file.

  Last resort (Pro failed twice):
    Add to prompt: "Think step by step. This is architecturally complex."
    /quit and relaunch for a fresh context window.

------------------------------------------------------------------------------
PART E -- CHECKPOINT REFERENCE
------------------------------------------------------------------------------

  Location: D:\staging\your-project\.checkpoints\ck01.md, ck02.md...

  Each checkpoint records:
    Session name and task ID
    Timestamp
    Files changed (auto-filled from git diff HEAD)
    What was completed (you paste SESSION-SUMMARY from Gemini)
    Decisions made
    Blockers
    What the next session should focus on

  On next session launch, the launcher reads the latest checkpoint and
  injects it at the top of the prompt. Gemini picks up exactly where the
  previous session stopped. You never lose context between sessions.

  Useful commands:
    Get-Content .checkpoints\ck01.md      (read a specific checkpoint)
    Get-ChildItem .checkpoints\            (list all checkpoints)

PART F -- TROUBLESHOOTING
------------------------------------------------------------------------------

PROBLEM: "cannot be loaded" or "running scripts is disabled"
FIX:     cd to the folder containing the .ps1 files, then run:
           Get-ChildItem -Filter *.ps1 | Unblock-File
           Set-ExecutionPolicy RemoteSigned -Scope CurrentUser

PROBLEM: "gemini: command not found" after install
FIX:     Close PowerShell and open a new window. npm global path only loads
         in a fresh shell. If still missing:
           npm install -g @google/gemini-cli

PROBLEM: gemini returns authentication error
FIX:     echo $env:GEMINI_API_KEY -- must not be empty. Redo Step A3.
         Or run: gemini auth   (this follows the Google OAuth flow)

PROBLEM: gemini quota exceeded
FIX:     Check your Pro plan at https://aistudio.google.com
         Contact your lead for a shared team key if needed.

PROBLEM: Setup script fails with "PRD.md not found"
FIX:     PRD.md must be inside $PROJECT_ROOT before running setup.
         Check that $PROJECT_ROOT in your setup script matches the folder
         where you saved PRD.md.

PROBLEM: Session launcher says "invalid session name"
FIX:     Run: .\your-project-sessions.ps1 -Session list
         If the module is missing, re-run the setup script with the
         corrected $MODULES list. It will regenerate the sessions file.

PROBLEM: Checkpoint not found / no resume block appearing
FIX:     Check that .checkpoints\ exists in $PROJECT_ROOT.
         If empty, it means no session has completed yet -- that is normal.
         The first session after setup creates ck01.md on exit.

PROBLEM: git push fails at session end
FIX:     Verify your remote: git remote -v
         Add remote if missing:
           git remote add origin https://github.com/YOUR/REPO.git
         Push may ask for credentials the first time -- that is normal.

PROBLEM: Gemini hallucinating API method names
FIX:     Add to prompt: "Use context7 to look up [library] before writing."

PROBLEM: MCP server not connecting
FIX:     Inside gemini: /mcp list
         If server shows disconnected: /mcp reload
         Verify npm global root matches settings.json:
           npm root -g

PROBLEM: superpowers not recognised
FIX:     Inside gemini:
           /extensions install https://github.com/obra/superpowers
         Then /quit and relaunch.

PROBLEM: Session looping or Gemini confused
FIX:     /quit -- relaunch with session launcher.
         Fresh context + checkpoint resume is always faster than debugging
         a confused session.
